# visualize-batch-outputs

Generate a single-file HTML reviewer UI for a pipeline run's `outputs.jsonl`.
Each line of the JSONL is treated as a serialized `RTMReviewState` (the shape
emitted by `tests/integration/test_pipeline.py::test_pipeline_parametrized`).

This skill is **read-only with respect to pipeline artifacts** — it only reads
the JSONL and writes an HTML file next to it. It does not mutate pipeline state
or call the LLM.

## When to invoke

- User asks to "visualize", "review in a browser", "open the HTML viewer", or
  "generate the output report" for a run directory.
- User references `logs/run-<ts>/outputs.jsonl` in the context of manual
  reviewer feedback.
- User is onboarding a reviewer and needs a shareable snapshot of a batch.

## Invocation

Default — write `viewer.html` next to the JSONL:

```bash
uv run python -m autoqa.viewer logs/run-<ts>/outputs.jsonl
```

Specify an output path:

```bash
uv run python -m autoqa.viewer logs/run-<ts>/outputs.jsonl \
  -o /tmp/review-2026-04-21.html
```

The skill folder also ships a thin wrapper script
(`.claude/skills/visualize-batch-outputs/generate_viewer.py`) that forwards to
the same entry point — use whichever form the user prefers.

**Note:** `viewer.html` is now auto-generated by the pytest session teardown
whenever `outputs.jsonl` has records (see `tests/conftest.py::jsonl_recorders`).
Only invoke this skill manually when the user wants to (re-)render a viewer
outside a test run, or target a custom output path.

Then tell the user: open the printed path in a browser (no server needed).

## Expected input shape

Each JSONL line must contain the serialized RTMReviewState keys:

- `requirement`: `{req_id, text}`
- `test_cases`: list of `{test_id, description, setup, steps, expectedResults}`
- `decomposed_requirement.decomposed_specifications`: list of
  `{spec_id, description, acceptance_criteria, rationale}`
- `test_suite.summary`: list of `SummarizedTestCase`
  (`{test_case_id, objective, verifies, protocol[], acceptance_criteria[], is_generated}`)
- `coverage_analysis`: list of `EvaluatedSpec`
  (`{spec_id, covered_exists, covered_by_test_cases[]}`).
  Each `covered_by_test_cases[i]` is `{test_case_id, dimensions[], rationale}`
  where `dimensions ⊆ {functional, negative, boundary}`. The per-TC `rationale`
  carries the V&V justification for that TC's dimension labels.
- `synthesized_assessment`: `{requirement, overall_verdict, mandatory_findings[5],
  comments, clarification_questions[]}`. Each `mandatory_findings[i]` is
  `{code, dimension, verdict, partial, rationale, cited_test_case_ids[], uncovered_spec_ids[]}`.
  `partial: true` (only valid when `verdict == "Yes"`) signals incomplete
  coverage that still passes SoP gating — drives Yellow rendering.

Fields are rendered defensively — a missing field shows as empty rather than
crashing the viewer.

## What the viewer shows

**Main window (two-panel layout):**

- **Left panel (information):**
  - **Requirement** — `req_id` chip + full text.
  - **Test Cases** — clickable list of `test_id`s. Click opens a modal with
    the raw test case (description / setup / steps / expected) plus the
    AI-summarized `SummarizedTestCase` (objective / verifies / protocol /
    acceptance_criteria).
  - **Coverage Assessment** — overall verdict badge (Yes = green / Yes-partial = yellow / No = orange)
    plus a bulleted table of the five mandatory findings M1-M5. Each row shows
    the dimension, a Yes/No/N-A chip (Yellow when `verdict=Yes` AND `partial=true`),
    the one-line rationale, and cited TC IDs or uncovered spec IDs. The overall
    verdict badge renders Yellow when `overall_verdict=Yes` AND any finding has
    `partial=true`. A "?" help icon next to the "Dimension" header opens a
    modal summarizing the M1-M5 criteria.
  - **"Decomposed specs & coverage analysis →" link** — opens a dialog with
    one row per decomposed spec, color-coded light green if `covered_exists`
    is true, light orange if false. Columns: spec_id, description,
    acceptance_criteria, covered flag, and covering TCs with dimension chips
    plus their per-TC V&V rationale (the per-TC `rationale` carries the
    justification — there is no longer a separate per-spec rationale field).
  - **Comments** — free-text gap clarifier from the synthesized assessment
    (rendered only when non-empty).
  - **Clarification questions** — bulleted list of targeted questions
    (rendered only when non-empty).

- **Right panel (input & control):**
  - **Rating** — radio buttons 1–5 for reviewer quality score.
  - **Notes** — free-text area for rationale or gaps the rubric didn't catch.
  - **Prev / Save & Next** — saves the current rating+notes to the browser's
    localStorage (keyed by `req_id`) and navigates. No server required.
  - **Progress** — `N / total` label.
  - **Export feedback JSON** (header button) — downloads all saved ratings+notes
    as a JSON blob for sharing.

Dialogs close via the **Close** button or by clicking the backdrop.

## Output & distribution

The generated HTML is a single file with inlined JSON data and vanilla JS —
no CDN calls, no network dependencies, safe to email or check into a review
artifact. Feedback lives in the browser's `localStorage` under the key
`visualize-batch-outputs/<run-dir-name>` until exported.

## Out of scope

- Recomputing or mutating any pipeline field. The viewer is a renderer.
- Re-running the LLM pipeline. Use the pytest integration tests for that.
- Multi-batch diffing or cross-run comparison. One run per viewer file.
- Server-side feedback persistence. Reviewer notes live in browser storage
  until exported via the header's "Export feedback JSON" button.
